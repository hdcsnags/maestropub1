# Vulnerability Disclosure Policy

## Reporting Security Issues

If you discover a security vulnerability in this project, please report it privately and do not open a public issue.

- Contact: security@example.com
- Subject line: `Security Disclosure - Session 74`
- Include:
  - A clear description of the issue
  - Steps to reproduce
  - Affected versions or commit hashes
  - Impact assessment
  - Any suggested remediation

## Response Expectations

We will aim to:

- Acknowledge receipt within 5 business days
- Triage and validate the report
- Provide status updates as appropriate
- Remediate confirmed issues in a reasonable timeframe based on severity and impact

## Coordinated Disclosure

Please allow reasonable time for investigation and remediation before any public disclosure. We support coordinated disclosure and appreciate responsible reporting.

## Scope

This policy applies to the code and configuration contained in this repository. Third-party services, dependencies, or infrastructure not maintained in this repository may be out of scope and should be reported to their respective maintainers.

## Safe Harbor

We will not pursue action against researchers who:

- Make a good-faith effort to avoid privacy violations, data destruction, and service disruption
- Only test against systems and assets in scope
- Do not exploit a vulnerability beyond what is necessary to demonstrate its existence
- Do not access, modify, or retain other users' data
- Report findings promptly and privately

## Non-Qualifying Reports

The following are generally not considered valid vulnerabilities by themselves unless they demonstrate a concrete security impact:

- Missing security headers without exploitability
- Version disclosure
- Self-XSS
- Best-practice recommendations without a demonstrated risk
- Reports affecting only outdated, unsupported environments

## Thank You

We appreciate the efforts of security researchers and the community in helping keep this project and its users safe.
